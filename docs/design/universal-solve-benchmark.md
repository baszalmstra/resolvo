# Universal solve benchmark

Status: in progress (2026-06-11). Numbers below are filled in as runs
complete; this header is removed when the campaign is done.

This report benchmarks `Solver::solve_universal` at scale on real
conda-forge repodata, characterizes its cost distribution against plain
(concrete-machine) solving on an identical problem corpus, diagnoses
outliers, and turns the results into a gating recommendation for pixi:
should universal solving be always on, opt-in, or auto-enabled only when a
manifest declares capability-dependent requirements?

## Methodology

### Setup

- resolvo branch `universal-solve-bench-fixes` (based on `universal-solve`,
  head `84b17610`), rattler branch `universal-solve-bench-fixes` (based on
  rattler `universal-solve`, head `367c07c0`).
- Snapshots of real conda-forge repodata (fetched 2026-06-11) for
  linux-64+noarch, osx-arm64+noarch and win-64+noarch, produced by
  rattler's extended `create-resolvo-snapshot` tool. Each snapshot is
  self-contained for universal solving: environment packages carry a
  marker, the pairwise relation table for all version sets targeting them
  (computed with the rattler oracle, including the archspec
  microarchitecture DAG) is embedded, and the simulated machine's concrete
  virtual-package candidates are stored as regular solvables. One snapshot
  file therefore drives both modes; resolvo stays free of conda knowledge.
- Benchmark driver: `tools/solve-snapshot` (release build), modes
  `concrete` (plain `solve` against the machine candidates) and
  `universal` (`solve_universal` against a JSON environment model,
  `tools/solve-snapshot/models/`).
- 1000 random problems per subdir, seed 0, 1 to 10 requirements each
  (weighted picks: packages 10x, version sets 1x, unions 1x). Environment
  packages and version sets targeting them are excluded from the random
  picks in every mode: real problems do not request virtual packages
  directly, and the exclusion is mode-independent so the corpus is
  identical across modes and rows join on the problem index.
- Timeout 60 s per problem. Runs strictly sequential on an otherwise idle
  Windows 11 machine. Universal runs additionally execute the independent
  verifier (`UniversalSolution::verify`) and a machine projection per
  problem, both outside the timed section.

### Environment models

The model is explicit and total: every region inside it must be solvable or
the universal solve fails with that region's conflict. Model literals use
ecosystem-exact bounds (the pairwise oracle cannot see ternary entailment).

- linux-64: `__glibc >=2.17,<3.0a0`; `__cuda` absent or `>=11,<14.0a0`;
  `__archspec` lineage includes `x86_64`; `__linux` and `__unix` present,
  unbounded. Simulated machine: glibc 2.35, cuda 12.4, archspec x86_64_v3,
  linux 6.8, unix 0.
- osx-arm64: `__osx >=11.0,<16.0a0`; `__archspec` lineage includes
  `aarch64`; `__unix` present. Machine: osx 14.5, archspec m1, unix 0.
- win-64: `__win` present, unbounded; `__archspec` lineage includes
  `x86_64`. Machine: win 10, archspec x86_64_v3. This is the near-trivial
  control model.
- Floor-overhead run (linux-64 corpus): a singleton model that pins every
  environment literal of the snapshot to its truth value on the simulated
  machine (124 unit clauses, generated by
  `tools/solve-snapshot/inspect_snapshot.py`). This measures pure
  universal-machinery overhead at one cell and answers "can pixi enable
  universal solving unconditionally for users with no capability needs".

### Semantics worth keeping in mind when reading the numbers

- `__archspec` literals use DAG lineage semantics ("the machine's
  microarchitecture lineage includes X"), deliberately unlike the
  exact-string matching of concrete `__archspec` candidates. In real
  repodata, microarch deps appear as `__archspec ==1 x86_64_vN` /
  `1.* x86_64_vN` (underscore spelling, matching the archspec DAG);
  osx-arm64+noarch repodata contains no microarch deps at all.
- Because the model is total, a problem whose packages require, say,
  `__glibc >=2.28` in every build fails the *whole* universal solve when
  the model floor is 2.17: the region [2.17, 2.28) is modeled but
  unsolvable. The concrete machine (glibc 2.35) solves the same problem.
  These "model-real unsolvable" rows are correct behavior, not solver
  bugs, and are counted separately below; in pixi the model floor would
  come from the manifest's system requirements, aligning the model with
  what the user actually supports.

## Results

### Pre-run fixes (found during smoke testing)

Three defects were found and fixed on `universal-solve-bench-fixes` before
the full campaign; the first two predate the universal work:

1. `SnapshotProvider::add_package_requirement` shadowed the snapshot's
   last version set (`Mapping::max` returns the highest id, not the
   count), so one version set per snapshot resolved to the wrong matcher.
2. `MappingIter` stopped after `len` slots and silently dropped every
   entry above the first holes of a sparse mapping; a deserialized
   snapshot is sparse, so iteration lost the tail (including all
   `--env-spec` version sets).
3. The environment witness search (coverage termination check) was a
   propagation-free exhaustive search. On a 196-cell solve it took 108 s
   of the 115 s total while the enumeration itself took under 4 s, and it
   ignores cancellation, so the 60 s timeout overshot freely. Adding unit
   propagation (false-first, ascending; identical lexicographically
   smallest witness, all witness tests unchanged) brought the same
   problem to 4.6 s end to end (25x).

### linux-64

CSVs: `bench-universal/full-linux-64-universal.csv`,
`bench-universal/full-linux-64-concrete.csv`.

|                      | universal | concrete |
|----------------------|-----------|----------|
| ok                   | 725       | 791      |
| unsolvable           | 250       | 202      |
| timeout (60 s)       | 25        | 7        |
| median duration      | 0.65 s    | 0.47 s   |
| p90                  | 5.1 s     | 2.0 s    |
| p95                  | 16.5 s    | 4.7 s    |
| p99                  | 60 s      | 44.8 s   |
| max                  | 78.1 s    | 60 s     |
| mean                 | 3.6 s     | 1.7 s    |

- **Paired ratio** (725 problems ok in both modes, joined on problem
  index): median **1.06x**, p90 2.10x, p95 4.46x, p99 25.8x, max 65.6x.
  At the median, universal solving costs the same as concrete solving;
  the overhead is concentrated in a tail of cell-explosion outliers.
- **Cells**: median 4, p95 44, max 360. 279 of 725 solutions (38%) are a
  single cell; 24 exceed 64 cells. Distinct environment literals in cell
  conditions: median 5, max 25. The worst paired ratios all coincide with
  triple-digit cell counts (e.g. 360 cells at 78 s vs 1.3 s concrete).
- **Verification**: the independent verifier accepts all 725 universal
  solutions (pairwise disjointness and model coverage). Machine
  projection produced a unique cell for every solution.
- **Outcome asymmetries**: 49 problems are unsolvable universally but
  solvable concretely. 39 of those fail in the old-glibc corner
  [2.17, 2.28) of the model (conda-forge's de-facto baseline has moved
  past 2.17, so many modern builds simply have no candidate there), and
  10 fail in the cuda-absent region (problems that transitively require
  cuda). Both are correct total-model semantics, not solver defects: the
  model claims those machines must be covered. The remaining 201
  universal-unsolvable problems are unsolvable concretely as well
  (broken or platform-foreign packages, e.g. msys2 packages whose epoch
  pins miss from linux repodata); the universal witness merely reports
  one (lexicographically baseline-most) failing region for them.
- 17 problems time out universally while solving concretely (between
  0.3 s and 28 s concrete); see the outlier section.

### osx-arm64

CSVs: `bench-universal/full-osx-arm64-universal.csv`,
`bench-universal/full-osx-arm64-concrete.csv`.

|                      | universal | concrete |
|----------------------|-----------|----------|
| ok                   | 606       | 609      |
| unsolvable           | 393       | 390      |
| timeout (60 s)       | 1         | 1        |
| median duration      | 0.167 s   | 0.167 s  |
| p95                  | 0.64 s    | 0.64 s   |
| p99                  | 1.47 s    | 1.44 s   |
| mean                 | 0.30 s    | 0.30 s   |

- **Paired ratio** (606 problems ok in both): median **1.00x**, p90
  1.05x, p95 1.08x, p99 1.13x, max 2.16x. Universal solving is free on
  this subdir.
- **Cells**: median 1 (563 of 606 solutions are a single cell), max 3.
  The reason is structural: osx-arm64+noarch repodata contains no
  `__archspec` microarch dependencies at all, `__unix` never
  discriminates, and only the `__osx` version axis occasionally splits.
  Without an expensive axis (cuda above all), enumeration degenerates to
  approximately one plain solve.
- The single timeout times out concretely too (an intrinsically hard
  problem, not a universal regression). All 606 solutions verify.
- 3 problems are universal-unsolvable but concrete-ok, all in old-`__osx`
  regions (packages whose builds all require `__osx >=12`/`>=13.3`
  while the model floor claims 11.0). 390 of 393 universal-unsolvable
  rows are concretely unsolvable as well.

### win-64

CSVs: `bench-universal/full-win-64-universal.csv`,
`bench-universal/full-win-64-concrete-merged.csv` (the concrete run was
interrupted: the machine went to idle sleep mid-run and a Windows Update
reboot followed on wake, so concrete row 821 is sleep-polluted, row 839 is
missing from the merged file, and rows 0-838 / 840-999 come from the
original and resumed runs; see the outlier section for the re-runs of 821
and 839).

|                      | universal | concrete |
|----------------------|-----------|----------|
| ok                   | 620       | 626      |
| unsolvable           | 378       | 371      |
| timeout (60 s)       | 2         | 2 (one is the sleep artifact) |
| median duration      | 0.34 s    | 0.34 s   |
| p95                  | 1.62 s    | 1.60 s   |
| p99                  | 6.4 s     | 5.3 s    |

- **Paired ratio** (619 problems ok in both): median **1.00x**, p90
  1.12x, p95 1.16x, p99 1.39x, max 8.79x. The near-trivial control model
  behaves as designed: universal solving is free here too.
- **Cells**: median 1 (579 of 620 single cell), max 3. All 620 solutions
  verify.
- 6 problems are universal-unsolvable but concrete-ok: 5 fail in the
  `__win < 10` corner (the model leaves `__win` unbounded, and modern
  builds transitively require `__win >=10`, e.g. via libboost), and one
  fails in a microarchitecture corner (`not skylake_avx512 AND not
  sapphirerapids`: a package whose builds all require AVX-512-level
  microarchitectures, which the `x86_64`-lineage model must cover). The
  same total-model floor semantics as on the other subdirs, on different
  axes.

### Floor overhead (linux-64 singleton model)

CSV: `bench-universal/full-linux-64-singleton.csv`. Universal mode on the
linux-64 corpus with every environment literal of the snapshot pinned to
its truth value on the simulated machine (124 unit model clauses): the
modeled space is exactly one machine, so this measures pure universal
machinery overhead.

- Outcomes are **identical** to the concrete run: 791 ok, 202 unsolvable,
  7 timeouts, with the same problem indexes in every class.
- Every solution is exactly 1 cell, and all 791 verify.
- **Paired ratio: median 0.96x, p90 1.00x, p95 1.02x, p99 1.06x, max
  1.20x.** The universal path is marginally *faster* than the concrete
  solve at the median: a pinned environment literal unit-propagates,
  which is cheaper than filtering and sorting the machine's virtual
  package candidate records that the concrete solve pays for.
- Conclusion: the enumeration machinery itself costs nothing. The entire
  universal overhead observed on linux-64 comes from genuinely covering
  more environments (more cells), never from the mechanism.
